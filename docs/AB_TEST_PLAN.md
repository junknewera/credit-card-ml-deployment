# План A/B-тестирования

## Постановка

Сравниваем две модели:

- Группа A (контроль): v1, CatBoostClassifier
- Группа B (тест): v2, RandomForestClassifier

Хотим понять, дает ли v2 улучшение по качеству предсказаний.

## Разделение трафика

50/50 по стабильному хешу от client_id - один клиент всегда попадает в одну группу, иначе данные смешаются при повторных обращениях. На уровне API роутинг делается через поле `model_version` в запросе. Распределять может вызывающий сервис или middleware.

## Продолжительность и выборка

2 недели - хватает чтобы покрыть рабочие и выходные дни и набрать нужный объем.

Минимальный размер группы считается по формуле для сравнения двух долей:

```
n = (Z_alpha/2 + Z_beta)^2 * (p1*(1-p1) + p2*(1-p2)) / (p1 - p2)^2
```

С параметрами Z_alpha/2 = 1.96 (значимость 0.05), Z_beta = 0.84 (мощность 0.80), p1 = 0.47, p2 = 0.50 получается примерно 3000 наблюдений на группу. При 200-300 запросах в день за две недели набирается 2800-4200 на каждую - должно хватить.

## Метрики

Основная - F1-score для класса дефолта. F1 учитывает и точность и полноту, что тут важно: нужно и ловить дефолтников, и не отклонять слишком много нормальных клиентов.

Дополнительная - Recall. Пропустить дефолтника (false negative) для банка обычно дороже ложной тревоги (false positive), потому что невозвращенный кредит это прямой убыток. При равном F1 лучше модель с Recall повыше.

## Статистический анализ

Двухвыборочный z-тест для сравнения долей. Выборки большие, метрика оценивается на уровне группы - z-тест тут подходит.

H0: F1(v1) = F1(v2), различий нет. H1: F1(v1) != F1(v2).

Для оценки разницы строим 95% доверительный интервал bootstrap-ом: 10000 выборок с возвращением из каждой группы, считаем разницу F1(v2) - F1(v1), берем 2.5-й и 97.5-й перцентили. Если интервал не включает 0, разница значима.

Тест успешен если F1(v2) > F1(v1) с p < 0.05, или если Recall(v2) значимо выше при не-снижении F1. Иначе оставляем v1.

## Поддержка на уровне API

`POST /predict` принимает `model_version` (по умолчанию v1), `POST /predict/v2` всегда идет на v2. В каждом ответе приходит model_version, так что при анализе всегда понятно какая модель отработала. request_id в логах позволяет проследить запрос.

Роутинг можно делать через nginx (upstream-веса), middleware в Flask или на стороне вызывающего сервиса.
