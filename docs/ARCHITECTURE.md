# Архитектура сервиса

## Монолит vs микросервисы

Сервис сделан как монолит. Тут особо нечего разделять: одна модель (точнее, две версии одной задачи) и один API. Трафик небольшой, отладка в одном процессе проще, деплой и откат тоже.

Микросервисы были бы оправданы, если бы появились модели с разными требованиями к ресурсам (допустим, одна на GPU, другая на CPU), или разные команды стали отвечать за разные части, или понадобилось масштабировать scoring отдельно от логирования. Пока ничего из этого нет.

## RabbitMQ (концепт)

RabbitMQ - брокер сообщений для асинхронного взаимодействия между компонентами.

В нашем случае он пригодился бы прежде всего для батч-предсказаний: клиент кидает в очередь список клиентов, воркер разбирает по одному, результаты складывает в базу, а API не блокируется. Ещё можно вынести логирование в отдельного consumer - тогда Flask не тратит время на запись в БД прямо в обработчике, и latency ответа падает. Ну и третий вариант - алертинг: если вероятность дефолта > 0.8, событие уходит в очередь и дальше в систему оповещений.

Схема простая: Flask - producer, RabbitMQ - брокер, один или несколько consumer-воркеров. Для Python есть библиотека `pika`.

## Логирование и мониторинг

API пишет структурированные JSON-логи на каждый запрос:

```json
{
  "request_id": "a1b2c3d4-...",
  "method": "POST",
  "path": "/predict",
  "status": 200,
  "latency_ms": 12.45
}
```

request_id генерируется сервисом или берется из заголовка X-Request-Id. Остальное - метод, путь, код ответа, время обработки.

В production такие логи обычно собирают через ELK (Elasticsearch + Logstash + Kibana) - хранение, парсинг и визуализация соответственно. Для метрик (RPS, latency, распределение предсказаний) подходит Prometheus + Grafana.

## DVC и MLflow (концепт)

В Git неудобно хранить модели по 50+ МБ и датасеты. DVC решает это: сами файлы уходят в удаленное хранилище (S3 или GCS), а в репозитории остаются .dvc-метафайлы с хешами. Для нашего проекта это model_v1.pkl, model_v2.pkl и исходный CSV. Любую версию модели можно воспроизвести через `dvc pull`.

MLflow закрывает другую задачу - отслеживание экспериментов. Гиперпараметры, метрики, артефакты - всё складывается в одно место, и можно быстро сравнить v1 с v2 по F1 или Recall. Реестр моделей полезен когда появляется staging -> production пайплайн: новая модель сначала проходит A/B-тест, и только потом едет в прод.

## Бизнес-метрики

F1, Precision, Recall - это про качество классификатора. Бизнесу они мало что говорят, нужны метрики привязанные к деньгам.

Первая - ожидаемые финансовые потери (Expected Financial Loss). Для каждого одобренного клиента: `P(default) * credit_exposure`, где P(default) берем из модели, credit_exposure - размер лимита. Суммируем по всем одобренным, получаем общий ожидаемый убыток. Модель которая снижает эту сумму при том же объеме выдач - лучше.

Вторая - доля одобренных заявок при фиксированном уровне риска. Ставим порог (допустим, 5% дефолтов) и смотрим сколько заявок проходит. Модель с лучшей разделяющей способностью пропускает больше заявок не превышая порог, а это уже выручка.
